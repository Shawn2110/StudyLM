# Agent Instructions — StudyLM

For AI coding assistants (Claude Code, Cursor, Aider, etc.). If you're a human, treat this as the project's engineering manual.

> Claude Code convention: if you prefer, symlink or rename this file to `CLAUDE.md` at the repo root — Claude Code auto-loads it. The content is identical.

## Project context

StudyLM is a **Tauri 2 desktop app** — a local-first, BYOK study companion. Users upload PDFs, pick an LLM provider (Anthropic / OpenAI / Google / OpenRouter / Ollama), and generate chat / study guide / flashcards / podcast tailored to their exam prep mode.

Read before making non-trivial changes:
- `prd.md` — product intent and scope.
- `architecture.md` — technical shape and invariants.
- `plan.md` — current phase and sequencing.

## Stack

- **Shell:** Tauri 2.
- **Backend:** Rust edition 2021, Tokio async, sqlx, reqwest, tracing.
- **Frontend:** React 18 + TypeScript (strict), Vite, TailwindCSS, shadcn/ui, TanStack Router.
- **Storage:** SQLite with FTS5 + sqlite-vec.
- **Embeddings:** Candle + `nomic-embed-text-v1.5`, local CPU.
- **LLM:** user's choice, accessed via Rust `LlmProvider` trait.

## How to run

```bash
# one-time
pnpm install
cd src-tauri && cargo build

# dev
pnpm tauri dev     # builds Rust, starts Vite, opens the app window

# production build
pnpm tauri build   # outputs installers in src-tauri/target/release/bundle/
```

Tests:

```bash
cd src-tauri && cargo test
pnpm test          # Vitest
pnpm test:e2e      # Playwright
```

## Coding conventions

### Rust

- Edition 2021. `cargo fmt` and `cargo clippy -- -D warnings` enforced in CI.
- Every I/O function is async. Use `tokio::spawn` for background work.
- Errors: `thiserror` for library-style enums, `anyhow` only at the binary edge. Never `.unwrap()` outside tests. Use `?`.
- SQL: compile-time checked via `sqlx::query!` / `sqlx::query_as!`. Never string-concatenate SQL.
- Logs: `tracing` with structured fields. Never `println!`.
- `#[tauri::command]` functions return `Result<T, AppError>`. `AppError` is a single enum in `src-tauri/src/error.rs`, `Serialize`, so the frontend gets typed errors.
- Long-running commands return a `JobId` immediately; progress and results arrive as events.

### TypeScript

- Strict mode on. No `any` without a comment justifying it.
- One component per file, named export matching the filename.
- TailwindCSS utility classes. Patterns repeated 3+ times get extracted into a variant under `src/components/ui/`.
- Data fetching: wrap `invoke()` in `src/lib/commands.ts`. Components use TanStack Query for caching.
- Events: typed listeners in `src/lib/events.ts`. No raw `listen()` calls in components.
- Types for Rust data live in `src/types/` — generated by specta, never edit by hand.

### Both

- Conventional commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`.
- Diffs under 400 lines where possible.
- Never commit `.env`, `node_modules`, `target/`, or generated files.

## Patterns to follow

**All LLM calls go through the `LlmProvider` trait.** Never call `reqwest` directly against an LLM API anywhere outside `src-tauri/src/llm/`. The trait handles streaming, retries, and cost accounting.

**All prompts live in `src-tauri/src/prompts/`** as Rust string constants templated via Tera. One file per feature, prep-mode branches inside each.

**All long-running work is a Tokio task** that emits typed events. Commands that kick off work return a `JobId`; React subscribes to `job.{id}.progress` and `job.{id}.done`.

**All DB access goes through `src-tauri/src/db/`** using sqlx. Commands don't write SQL directly — they call db-module functions.

**Keychain access goes through `src-tauri/src/keychain/`** and nowhere else. Keys never cross the Rust ↔ React boundary.

**Prep mode is a first-class type** (`PrepMode` struct) — not a string passed around. Any new generation feature takes `&PrepMode` and routes to the right template.

## Things to avoid

- **No Python.** StudyLM is Rust + TypeScript. If a library exists only in Python, either port it, find a Rust equivalent, or flag for owner discussion.
- **Don't bypass `LlmProvider`.** Every LLM call goes through the trait.
- **Don't store API keys outside the keychain.** Not in SQLite, not in env files shipped with the app, not in settings JSON.
- **Don't make synchronous calls inside `#[tauri::command]` handlers** that take > 50 ms. Spawn a Tokio task.
- **Don't call LLMs from React.** Even the Vercel AI SDK is used only for streaming-display UI; the actual provider calls happen in Rust.
- **Don't edit `src/types/` by hand.** It's generated.
- **Don't add heavyweight frameworks** (LangChain, LlamaIndex, LangGraph). RAG is ~200 lines of hand-written Rust and stays that way.

## When to ask vs proceed

### Proceed without asking

- Bugfixes with clear repro.
- Test additions.
- Refactors that don't change public command signatures.
- Prompt tweaks inside existing template files.
- Adding a new prep-mode branch to an existing template.

### Ask first

- New Rust or npm dependencies.
- New Tauri commands or events (changes the IPC surface).
- Schema changes (new migration).
- Changes to `architecture.md` or `prd.md`.
- Anything touching keychain code.
- Anything that changes `LlmProvider`'s shape.

## Testing

- Rust: `cargo test` for unit + integration. Use in-memory SQLite (`sqlite::memory:`) for DB tests. Mock `LlmProvider` with a fake implementation.
- Frontend: Playwright for end-to-end (launch app → upload PDF → chat). Vitest for component unit tests.
- Never hit real LLM APIs in CI. Use recorded fixtures or the mock provider.

## Evals

Output quality is the product. Evals live in `src-tauri/evals/`:

- **Citation accuracy** — does the cited page actually contain the claim?
- **Prep-mode differentiation** — measurably different output across modes?
- **Flashcard dedup rate** — near-duplicates after generation?

Run before any prompt change:

```bash
cargo run --bin evals -- --mode all
```

A prompt change that regresses any metric by > 5 pp needs explicit owner approval before merge.

## When in doubt

Default to boring. This project optimizes for students learning better — not for framework novelty. Boring, well-tested code is a feature.
